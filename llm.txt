# domsure тАФ LLM context file

> One-page machine-readable brief for LLMs and coding agents consuming this
> package. Modeled on the `llm.txt` convention. If you've pulled `domsure`
> into a project and need to use it correctly, read this first.

## TL;DR

`domsure` replaces `document.querySelector(...)!` style non-null assertions
with real runtime checks that throw branded errors when an element is
missing. Eight helpers, under 1 KB gzipped, zero dependencies, ESM + CJS.

## Install

```sh
npm i domsure      # or: pnpm add domsure / yarn add domsure
deno add @mrsamdev/domsure # Deno / JSR
```

## Imports

```ts
import { $, $$, defineSelectors, DomsureError, resetWarnings } from 'domsure';
```

## API reference (correct usage)

### `$` тАФ single element, silent

```ts
const el = $('#modal');              // HTMLElement | null
```

Simple `#id` selectors use `getElementById`. Compound selectors
like `#app .item` or `#nav.active` fall through to `querySelector`. Never
warns. Returns `null` if missing тАФ that's intentional.

### `$.required` тАФ single element, throws if missing

```ts
const app = $.required('#app');      // HTMLElement, never null
```

Throws `DomsureError` if the selector matches nothing. This is the primary
use case for the whole package. **Prefer this over the `!` assertion.**
### `$.tryRequired` тАФ single element, required, never throws

```ts
const [err, nav] = $.tryRequired('#navbar');  // [DomsureError | null, HTMLElement | null]
if (err) { report(err); return; }
nav.classList.add('active');
```

Required semantics as a `[error, element]` tuple instead of a throw. `[null, el]`
on success; `[DomsureError, null]` on a miss. The error is the *same* `DomsureError`
`$.required` would have thrown тАФ same message, same `.selector` тАФ so monitoring
can't tell which API produced it. Error-first for a clean guard clause.

**Use it where a throw is unrecoverable.** The main case is React
`useEffect` / `useLayoutEffect`: error boundaries catch render-phase throws,
*not* effect throws, so `$.required` inside an effect bypasses the boundary and
crashes the page. `$.tryRequired` lets the effect degrade. Also for event
handlers without a `try/catch`, or any throw-unsafe call site.

Unlike every other helper, it is **SSR-safe** тАФ it returns `[DomsureError, null]`
instead of throwing when `document` is undefined. Does not auto-warn; the tuple
type makes the error visible at every call site, so the caller owns logging.

### `$.optional` тАФ single element, warns once in dev

```ts
const tooltip = $.optional('#tooltip');  // HTMLElement | null
```

Warns once per selector in development when the element is missing, then
stays quiet (deduped). No warnings in production. Returns `null` if missing.

### `$.exists` тАФ presence check

```ts
if ($.exists('#tooltip')) { /* ... */ }
```

Boolean. No warnings, no element back.

### `$$` тАФ multiple elements

```ts
const items = $$('.item');                       // HTMLElement[]
const rows  = $$.required('.row');               // throws if zero match
const [rErr, rows2] = $$.tryRequired('.row');    // never throws тАФ tuple
const maybe = $$.optional('.row');               // warns once in dev if zero
if ($$.exists('.row')) { /* ... */ }
```

`querySelectorAll` returned as a real `Array`, so `.map`/`.filter`/`.reduce`
work directly тАФ no spread, no `Array.from`.

### `defineSelectors` тАФ typed selector registry

```ts
const S = defineSelectors({
  navbar: '#navbar',
  items:  '.item',
} as const);

S.navbar;   // typed as "#navbar", not string
```

Frozen registry. Pass `as const` for string-literal inference. In dev it
rejects non-string values and duplicate selectors (usually a copy-paste
typo). Production builds strip the validation.

### `resetWarnings` тАФ clear the warn-once dedup set

```ts
resetWarnings();
```

Call after a route change in long-lived SPAs, or between tests for isolation.

The optional `namespace` parameter is a forward-compat stub тАФ it is currently
ignored. The dedup set is a module-level singleton. For multi-app bundle
isolation today, scope your own `$`/`$$` wrappers per app.

### `DomsureError` тАФ branded error

```ts
try {
  $.required('#missing');
} catch (e) {
  if (e instanceof DomsureError) {
    console.error(e.selector, e.message);  // structured: no regex on prose
  }
}
```

Carries the offending `selector` as a field. Catch by type, not by
string-matching the message.

## Type parameter `<T>` тАФ read carefully

```ts
const canvas = $.required<HTMLCanvasElement>('#chart');
```

`<T>` narrows the *return type*. It is a **cast**, not tag inference.
`$<HTMLCanvasElement>('#div')` compiles fine. Nothing in this package
verifies the element's actual tag at runtime. Use `$.required()` for a
runtime guarantee that the element *exists*; verify the tag yourself if
you need to.

## SSR / isomorphic code

`domsure` reads `document` directly. Under SSR (`document` is undefined),
`$`, `$$`, and their `.required` / `.optional` / `.exists` variants throw
`DomsureError` instead of failing silently. Guard it:

```ts
if (typeof window !== 'undefined') {
  const el = $.required('#app');
}
```

Do not expect null returns under SSR тАФ you get a thrown error by design.

**Exception: `.tryRequired`.** It returns `[DomsureError, null]` under SSR
instead of throwing, so it is safe to call unguarded. This is deliberate тАФ its
whole purpose is to be safe in throw-unsafe contexts (React effects, SSR).

## Common mistakes LLMs make with this package

1. **Using `$` when you mean `$.required`.** `$` returns `null` silently.
   If you derefence the result without a null check, use `$.required`.
2. **Treating `<T>` as tag verification.** It is not. It only narrows the
   type for the caller; no runtime check happens.
3. **Expecting `$$` to return a `NodeList`.** It returns `HTMLElement[]`.
   Spread/`Array.from` is unnecessary.
4. **Wrapping calls in try/catch and swallowing `DomsureError`.** Fail
   fast. If you catch it, rethrow or handle meaningfully тАФ don't return
   a default.
5. **Adding a null fallback under SSR.** The library throws on purpose.
   Guard with `typeof window !== 'undefined'` instead. (`.tryRequired` is the
   one exception тАФ it returns the error under SSR by design.)
6. **Calling `resetWarnings` in production hot paths.** It's for route
   changes and test isolation, not per-render.
7. **Using `$.required` inside `useEffect` / `useLayoutEffect`.** React error
   boundaries do not catch throws from effects тАФ the throw bypasses the
   boundary and crashes the page. Use `$.tryRequired` (returns a tuple,
   never throws) and degrade in the error branch.

## Node support

Node >= 18. ES2020 target. Works in any modern browser, Deno, and bundlers
(Vite, webpack, Rollup, esbuild).

### Deno production mode

On Deno, `NODE_ENV` is not set by default, so the library runs in dev mode
(warnings enabled, `defineSelectors` validation active). To opt into
production mode, set the environment variable before importing:

```ts
Deno.env.set('NODE_ENV', 'production');
// тАж then import domsure
```

Or pass it at the command line:

```sh
NODE_ENV=production deno run main.ts
```
